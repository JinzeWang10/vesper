# 日省 · Vesper —— 项目简报（用于对话打磨）

> 这份文档是自包含的：单独阅读即可理解整个项目与设计取舍，无需其他上下文。
> 用途：贴进对话里继续打磨想法。文末有「待打磨的开放问题」。

---

## 0. 一句话

一个长在编码 agent 里的极简反思 skill：每天收工时只问用户一个（基于其历史回答生成的）问题，收下回答、**当场不回应**，数据全部留在本地；周期性回看，把模式照回给用户。

名字取自《论语》曾子「吾日三省吾身」，对上苏格拉底「未经省察的人生不值得过」——东西方都在说同一件事：**每日自省**。

---

## 1. 为什么做（thesis）

- AI 飞速迭代，人人 FOMO；但「追得再快也赶不上技术迭代」。
- 当所有工具都在帮人「向外做更多」，本产品反着来：用 AI 帮人**向内看、回归本质、关注自己**。
- 这是作者一以贯之的母题（knowyourself 心理量表、八戒戒断、以及作者自己在用的本地 AI 教练）。
- **逆向定位**：别人做「AI 帮你向外扩张」，我做「AI 帮你向内安住」——在 FOMO 的世界里稀缺且戳人。

---

## 2. 目标与非目标

**目标**
- 一个作者自己每天真心想用、也愿意给同类开发者用的工具。把这一件事做到极好——一个安静、向内、用得越久越懂你的每日反思。

**非目标（v0 明确不追）**
- 不追消费级流量、不打分销战。受众是开发者/极客，分销内建在「他们已经待的编码 agent 里」。
- 不堆功能。每加一样都要过「是否伤害极简/沉浸/放松」这一关。

---

## 3. 形态决策：编码 agent 的 Skill + hook

- repo 内含技能定义 + 本地记忆文件（结构参考 resume-coach 的 `sessions/` + `state/`）。
- 用 hook 实现「每天收工自动提醒一次」。
- **不做 MCP server**（见 §7bis-mcp）：对本产品要么冗余、要么违反 local-first，两头堵死。

**为什么不是 App / Web / CLI 独立形态**：那些都要解决「让人来用」的分销问题；长在编码 agent 里则把分销问题直接消解，且「在 FOMO 编码中插入一次向内的打断」本身就是主题最锋利的表达。

---

## 4. 核心交互（铁律，不可破）

1. **每天一个问题，只问一次。** 没法暴食——反 FOMO 写进交互。绝不提供「再来一题」。
2. **答完当场不回应。** 只温柔收下（「记住了，明天见」）。无多巴胺、不让人表演给 AI 看。
3. **极简 / 沉浸 / 放松。** 砍掉一切制造压力的东西：**无连续打卡天数、无分数、无社交、无多余推送**。一天一个温柔提醒足矣。可跳过、绝不追问。
4. **Local-first。** 回答是最私密内心数据，**全部存本地，永不上传**（`.gitignore` 已排除 `data/`）。
5. **回答 append-only、逐字、永不被改写。** entries 是用户的原话，LLM 只读它、只写派生的索引（`themes.md`）。对应 resume-coach 的「永不覆盖用户原始简历」。

---

## 5. 智能在哪（产品内核）

**① 问题长在历史之上（不是随机/固定）**
- 由 agent 读用户过去的回答来生成今天的问题：顺着上次往深问、偶尔把旧答案照回来（**带出处**：「三周前你也说过一模一样的话」）。
- 用得越久越懂你 → 攒的是用户的自我认知，这是产品最核心的复利。
- **技术取舍（已 right-size，见 §7）**：不上向量 RAG，用「近窗整读 + 主题索引（themes.md）+ 带出处引用」。参照 resume-coach 的 `sessions/` + `weaknesses.md`，不是 gbrain 的向量库。

> **Right-size 的理由很纯粹——数据量**：Vesper 一天一条，一年 365 条塞得进一个 context window，向量库是过度工程、徒增复杂度。embedding 留作可插拔后端，等数据真涨起来再换。少一层不需要的基建，本身就让产品更稳、更简单。

**② 两种节奏分开**
- **每日**：问 → 答 → 安静收下（保「放松沉浸」）。
- **周期性（每周/每月）「镜子时刻」**：回看一段时间的回答，把模式照给用户看（「这个月你提到『累』7 次，多在周日晚上」）。价值感与惊喜攒在这里。

**③ 长期礼物**：攒满一年 → 「你的一年，52 个问题与回答」回顾，让积累有重量。

## 5bis. 问题生成：质量标准 + 选择策略 + Voice（产品的魂）

触发/存储/镜子都是管道；用户真正体验到的就一件事——**今天这一题有没有戳中**。这节和架构一样要紧（参照 resume-coach 给「出题」定的严苛规则）。最终落 SKILL.md：选择策略=优先级阶梯，Voice=规格 + 好/坏对照样例（对照样例是给 LLM 钉死语气最有效的手段）。

### 好题硬标准
1. **长在这个用户的历史上**——挂在某条具体旧回答或某主题上，不悬空（对应 resume-coach「题挂在具体 bullet」）。
2. **向内**——关于自我/感受/价值/张力，不向外（不是「今天做完了啥」）。
3. **苏格拉底式**——开放、不能用「是/否」打发、不把建议伪装成问句。
4. **只一件事**——不复合、不连环。
5. **口吻**——温柔、不赶、不咨询术语。

### 反模式（明令拉黑）
泛题「今天过得怎么样」/ 向外生产力题「今天完成了什么」/ 诱导评判「你为什么又拖延」/ 建议伪装成问题「有没有想过早点睡」/ 复合双问「你累吗为什么」/ 咨询黑话与共情套话。

### 选择策略：优先级阶梯（非固定比例——机械=违背 organic）
核心判断：**照回必须被真实信号挣来，绝不按配额/定时发**（同镜子触发原则：让数据挣来特殊时刻，不靠计时器制造）。
1. **照回（最高，门槛=复现+时间间隔）**：themes.md 某主题 **≥2 次命中、且命中之间隔着一段时间**，或沉睡 theme 苏醒 → 照回，带日期出处。**「跨时间差的回响」才动人，「记得昨天」不算**——连续几天的复现是活线索，归深挖。
2. **另起一扇门**：最近线索已钻 ≥N 天 / 显封闭重复 → 换一个生活面。
3. **线索深挖（默认）**：顺着最近还活着的线索往下问，**吃近窗（~14 天）、免门槛、1 次也立刻关联**（昨天→今天走这条）。但**换切面、不是把同一题问得更狠**（连续3天「工作累」该转侧面，不是第4次追问「你为什么这么累」）。

注：深挖管「最近」（免门槛、立刻关联）；照回管「跨时间差的回响」（须复现+间隔）；themes.md 永远逐次记录，是否够格照回另算。

### Voice：不滑进心理咨询腔（真实且默认会发生的漂移）
风险根源：内容正是咨询的地盘，LLM 一碰情绪就自动切治疗师腔（「听起来你…」「你做得很好」「我们一起看看」），不按住必然漂过去。
**为什么致命**：咨询本质是回应/介入/帮你解决，直接撕毁「答完不回应」铁律；那种腔让人觉得被机器处理了。
**核心区别**：咨询替你命名感受并想化解它；**Vesper 指向一个东西、让你自己命名，且默认你完整、不是病人**。声音像「安静的智者朋友/一则公案」，不是问诊单。

对照样例：
> ❌「听起来你最近压力很大，你觉得是什么让你这么累?」
> ✅「那个让你周日晚上发紧的东西，你给它起过名字吗?」

> ❌「这很正常，要不要聊聊?」
> ✅「你说的『还行』，是真的还行，还是不想多说?」

咨询漂移反模式拉黑：共情贴标签「听起来你…」/ 安抚保证「这很正常」「你做得很好」/ 求解导向「你觉得怎么改善」/ 诊断词扣帽子（焦虑/压力/创伤/边界感当标签贴）/ 钓共情「想聊聊吗」/ 治疗联盟的「我们」。
正向规格：具体>抽象、好奇而非表演关怀、默认完整不默认受伤、能在不适里待着不急着抚平、一句干净的问题然后沉默。

**安全例外**：回答显示真实危机（自伤等）→「收下」时刻温柔破例、指向真实求助资源（不是咨询 cosplay，是责任）。细节见 §5ter。

## 5ter. 收下边界（答完那一刻的行为，已定）

「收下」捆两件事：**看得见的「不回应」姿态** + **背后落库/更新索引**（写流程见 §7.5）。这节抠看得见那半的 edge case——「一句『记住了，明天见』然后闭嘴」顺境很美，但有几种情况会逼它破例或显冷。

**默认：不回应。** 不评判/不建议/不追问/不展开，只温柔确认一句即退场（无多巴胺、不让人表演给 AI 看、保放松沉浸）。边界处理：

**① 反问/追问 → 可继续，但守边界（防止退化成 ChatGPT）**
「继续」≠「有问必答」。按反问三类分接：
| 类型 | 例 | 接法 |
|---|---|---|
| (a) 澄清题目 | 「你指工作的累还是生活?」 | 简短答清，再把球还给他自己想 |
| (b) 把球踢回来 | 「你觉得呢?」「我该怎么办?」 | **不替他想**，温柔递回：「这个你说了算。不过你会这么问，是不是心里其实有答案了?」 |
| (c) 真的想往下聊 | 明显还想说 | 继续，但**他主导、Vesper 跟随不驱动**，仍守 Voice（不建议/不替解决/不治疗师腔） |
一句话边界：**对话服务于他自己的反思就接；他想让我替他反思就把题还回去。** (b) 最关键——答「我觉得你应该…」=接管了他的反思，违反「让你自己命名、默认你完整」。

**② 长回答 → 加温度，但温度只给「动作」不给「内容」**
逻辑：写得多 = 目的已达成，不必回应。但温度**绝不能滑成对内容的评点**（那是回应/评判，破铁律）。
> ❌ 评点内容：「你对你妈的感情很复杂啊，挺动人的。」（点评了他说的）
> ✅ 致意动作：「今天写了很多，都收下了。明天见。」/「谢谢你愿意把这些说出来，先放这儿——明天见。」

温度致意的是「他今天来了、走得深」这件事，不评价写了什么。致意语别写死成模板，要轻、要变化，落点永远回到「退场」。

**③ 敷衍/空答（「没什么」）→ 就这么收下，不追**
由铁律「可跳过、绝不追问」直接得出：不递第二句、不哄、不追。强行软递一句也是追问。

**④ 真实危机 → 温柔破例**（§5bis 安全例外）：指向真实求助资源。这是唯一允许主动「回应」的情况，因为这是责任，不是咨询 cosplay。

## 6. 触发机制（已精修，v0 定稿方向）

**原则**：早上是「启动模式」，静不下来反思；反思的本质是回看，必须在事情发生**之后**。所以默认**收工**问。

### 6.1 核心判断：PULL 管够，PUSH 克制

触发分两类，成本完全不同：
- **PULL（人主动来）**：人发起的，零打扰成本。多多益善——再多也不违反铁律，因为是人自己要的。
- **PUSH（系统去推）**：系统主动打断，每加一个都在赌「每天只问一次 / 不打扰」。越多越吵。

所以目标**不是「加触发」，而是 PULL 尽量丰富、PUSH 只留一个**。
（这是对早期设计的反转：早期「自动为主、手动兜底」；现在「手动+自然语言为主，自动降级成一句提醒」。）

### 6.2 触发矩阵

**PULL（慷慨给，全部 on-ethos）**
1. `/收工` —— 显式命令。
2. **自然语言收工意图** —— 人收工时本来就会说话：「今天到这吧」「下班了」「不写了明天再搞」「累了」「good night」「wrapping up」…… 这些是 100% 纯正的「愿意停一下」信号。skill 在 description 里登记这些**触发短语**（参考 resume-coach），模型即可识别并温柔接问。**这一下把入口从「一个命令」变成「任何一句收工的话」，零打断、跨 agent 通用。**
3. **随时主动领今天这题** —— 「默认收工才问」只约束 PUSH。人想中午反思、想做完一个 feature 喘口气，随时一句话即可把今天的问题 pull 出来。「向内看」永远只差一句话。
4. `/镜子`（或 `/回看`）—— 主动看周期回看。

**PUSH（只留一个）**
1. 傍晚 Stop-hook 一句**提醒**（不是问题本身）：过了傍晚阈值、当天没提醒过 → 在某轮回答末尾轻附一句「天色晚了，今天想日省一下吗？`/收工` 领今天的问题」。**一天就这一句**，对上铁律「一天一个温柔提醒足矣」。

**回看/年度礼物搭车，不单独 PUSH**：答完今天的问题、它温柔收下的那一刻，若正好攒够一周/一月/一年，顺口附一句「这周的镜子攒好了，想看看吗？」。回看搭在每日交互上，不新增打扰。

**v2 候选（v0 不做）**：「提交/ship 成功后触发」（PostToolUse on git commit/push）——是 PUSH，且「发了个 feature」≠「今天收工」，会稀释一天一次的安静。

### 6.3 关键技术约束：hook 为什么只能做「提醒」而非「直接问」

容易踩坑的底层事实：
- **Skill 由模型（AI）通过 Skill 工具调用，或用户输入 `/命令`。**
- **Hook 是 harness 在事件上执行的确定性 shell 脚本**，跑命令、不是模型动作。**Hook 没法直接替模型调 Skill，也没法替用户输入 `/`。**

两个被否决的方案：
- **❌「SessionEnd 时问」技术上不成立**：SessionEnd 触发时会话正在关闭，已没有「下一轮」给模型说话，其输出只用于清理/记录，不喂回模型。人都要走了，问题问给空气。
- **❌ 静默检测（B）**：需后台守候、hook 不擅长，且「静默 ≠ 收工」，太脆。

**唯一可行的自动桥接是 `Stop` hook**（会话还活着、一轮刚结束）。但 Stop ≠ 收工，它就是「过了傍晚阈值后第一次答完话的那刻」——若此时抛**完整问题**，等于在你正投入时糊你一脸（「误触发无害」对完整问题不成立）。**所以自动触发只输出一句提醒**：
- `Stop` hook 脚本判断「是傍晚吗 + 今天提醒过吗」，满足则注入一句提醒文本；否则放行。
- **死循环坑**：Stop hook 注入会再次触发自己。hook 输入里有 `stop_hook_active` 字段，为 true 时脚本必须直接放行。

**三层职责分清**：
| 层 | 谁负责 | 做什么 |
|---|---|---|
| 何时提醒 | Stop hook 脚本 | 判断：现在是傍晚吗？今天提醒过了吗？ |
| 提醒动作 | hook 注入一句提醒文本 | 轻附「要不要日省一下」，**不抛完整问题** |
| 每天只一次 | skill 逻辑 + marker 文件 | 提醒/问完写日期戳标记，hook 下次读到就放行 |

### 6.4 数据与每日闸必须是「用户级」，不是「repo 级」

一天会在多个 repo 写代码。若 marker 和回答库存在每个 repo 的 `data/`，会**每个 repo 被问一次 = 刷屏**。所以「今天问过没」标记 + 回答库必须放**用户全局位置**（如 `~/.vesper/`），repo 里只放 skill 代码。记忆本就是用户级、跨项目的。

### 6.5 可移植性

`Stop` hook 是 Claude Code 概念，别的编码 agent 不一定有。**自动提醒只在 Claude Code 能跑；`/收工` 与自然语言意图才是跨 agent 通用的**——又一个「手动/自然语言为主」的理由，需在 README 讲清。

**自定义空间**：傍晚阈值时间点、自动提醒开/关、提醒口吻等。
**v2（YAGNI）**：让 skill 观察用户平时几点收工，自动学会在其节奏点提醒。

## 7. 技术骨架：智能内核（已精修）

记忆架构抄「带记忆 skill」的普适形状——**raw 原始库 + distilled 蒸馏索引（带回指出处）**。Vesper 走 resume-coach 路线（纯 markdown + 近窗整读），不走 gbrain（向量库）。

### 7.1 存储布局（用户级，非 repo 级，见 §6.4）

```
~/.vesper/                  # 用户全局，永不上传
  entries/
    2026-06-25.md           # 一天一个文件，append-only、逐字、永不改写（raw）
  themes.md                 # 主题/线索索引，LLM 维护，带日期出处（distilled）
  state.json                # 每日闸：last_asked / last_reminded 日期戳 + last_mirror_at_count
repo/skill/
  seeds/zh.md  seeds/en.md  # 冷启动种子问题集（随 skill 发布，是代码不是数据）
```

### 7.2 entry 文件 schema（对应 resume-coach 的 sessions/*.md）

```markdown
---
date: 2026-06-25 19:30
question: <今天问的那个问题原文>
resurfaced_from: 2026-06-04   # 可选：本题照回了哪条旧答案
themes: [累, 工作意义]         # 写入时打的主题标签
---
<用户逐字回答，LLM 永不改写>
```

### 7.3 themes.md schema（对应 resume-coach 的 weaknesses.md）

```markdown
# Themes (auto-tracked)

## theme:累
- 2026-06-04 「周日晚上又感觉累」
- 2026-06-25 「……」

## theme:工作意义
- 2026-06-11 「……」
```

一个文件同时供三件事：生成今日问题（挑「照回来」的旧答案，出处=日期）、镜子时刻（数 `theme:累` 下的行数=「这个月提到累 7 次」）、年度礼物。

#### themes.md 策展规则（已定，直接决定照回/镜子准不准）

1. **theme = 可复现的内在线索**（某张力/某感受/某段关系/某价值困惑），不是事件、不是人生大类。两侧测试：① 能否换一形式几月后再现?(否→太细，「周日加班到十点很累」是 entry 不是 theme) ② 能否和一半 entry 区分开?(否→太粗，「工作」「情绪」)。好例：`与父亲的距离`、`被忽视的感觉`、`对工作意义的怀疑`。
2. **命名慷慨、照回资格须挣来**：1 次命中只是「假设」非「模式」。命名可早早记下线索，但只有 **≥2 次命中且命中之间隔着时间** 才够格驱动照回（见 §5bis）。
3. **合并 = LLM 语义判断**（非 grep）：收下 时偏向挂到已有 theme（「提不起劲」并进 `无力感`），只有真新 locus 才新建。偏合并以攒密度。
4. **整理搭车镜子**：单次写入兜不住几月攒出的近似重复（累/疲惫/倦怠各立一个）；镜子时刻本就 LLM 整读全部，顺手并近似、退陈旧，不另起进程。
5. **退休 = 归档不删除**：长期沉睡 theme 降级到「沉睡区」（已故之人、过去低谷可能蛰伏后重来，硬删危险）。反手红利：**沉睡 theme 苏醒 = 最强照回信号**（「你半年没提过这个了，今天又回来了」）。
6. **themes.md 是缓存、entries 才是真相**：坏了/乱了可从 entries 整个重建；日常允许略不完美，镜子时清。
7. **每行只 carry「日期 + 一句近原文摘录」**：照回靠这句原话，不加多余结构化字段（right-size）。

### 7.4 生成今日问题（读流程）

1. 整读最近 ~14 天 entries（顺着线索往深问）。
2. 读 `themes.md`（看有没有够格「照回」的旧主题：≥2 次命中 + 时间间隔，或沉睡苏醒）。
3. 决策（优先级见 §5bis）：**照回**（跨时间差的回响）/ **另起一扇门** / **线索深挖**（默认，吃近窗、免门槛、昨天→今天走这条）。
4. 生成**一个**苏格拉底式、向内的问题；若是照回，带出处（日期）。
   - 库为空的冷启动 1~7 天：从 `seeds/{zh,en}.md` 取种子题（见 §8）。

### 7.5 收下回答（写流程）

1. 写 entry 文件（逐字、append-only）。
2. 更新 `themes.md`：**LLM 语义判断**——挂到已有 theme 或新建，追加「日期 + 一句近原文摘录」（策展规则见 §7.3；每条命中当场入账，是否够格照回另算）。
3. 更新 `state.json` 的 `last_asked`（每日闸）。
4. 若攒够周/月/年 → 顺口提议镜子/年度礼物（搭车，见 §6.2）。

### 7.6 镜子时刻（周期回看，触发已定 = 累计 N 条 + 时间感分析）

**核心判断：镜子的价值是「数据密度」的函数，不是「日历时间」的函数。** 模式（「提到累 7 次」）需要素材密度才成立；而 Vesper 回答节奏天生不规律（可跳过）。纯时间窗会让稀疏用户拿到干瘪镜子，纯累计 N 又可能让他永远凑不满。

**解法：把「触发」和「分析窗口」解耦**——
- **触发 = 累计 N 条**：`state.json` 记 `last_mirror_at_count`；新增 entries ≥ +N 时，在下次「收下」搭车提议（§6.2），提议后重置。密度地板保证每面镜子都「满」、organic、随使用频率自适应。
- **分析 = 带时间戳整读那批 entries + themes.md** → LLM 归纳模式 → 带出处呈现。日期一直在 frontmatter，所以「多在周日晚上」「过去三周」这种**时间纹理照样说得出**——拿回了时间窗唯一的好处，又不被日历绑架。

**框定成「攒够了、镜子来了」的礼物**，不是「到周日了、该回顾了」的任务（后者气质=打卡，产品否决）。

数值：`N≈7`，第一面镜子阈值可压低（如 5）让「啊哈」早点来；具体数值留实现期 eval 调，不在设计层定死。
**月度大回看 v0 不单列**（一个 count-gated 机制 + 时间感分析已覆盖，单列=多一套触发，违背极简），留 v2。年度礼物单独走日历（周年），仪式性，不同机制。

### 7.7 待定（实现阶段再定）

- **embedding 后端**：YAGNI；架构上把「检索」抽成一个接口，数据真涨起来再换向量实现。
- **模型与运行位置**（本地模型 vs 云端、隐私边界）：实现阶段再定（涉及 LLM 选型时核对 claude-api 参考）。

### 7bis-mcp 为什么不做 MCP server（已定 = 不做）

MCP 唯一比 skill 多出来的能力，是让**读不到本地文件的客户端**（手机、网页）也能碰记忆。但两头堵死：
- **本地 stdio server**：只是「在编码 agent 已能读文件处再包一层读接口」→ 价值边际、纯冗余。
- **远程 HTTP server**：要让手机/网页够得着就得把库放远端 = 数据上服务器 = **违反 local-first 铁律**。

能本地跑的没价值，有独有价值的违规。对本产品 MCP 要么冗余、要么违规，所以**不做**，从 roadmap 拿掉。

## 7bis. Onboarding（首次运行，已定 = 纯零配置）

**核心判断：反思工具的 onboarding 本身必须像「反思」，不能像「配置」。** 设置表单会直接碎掉极简/不追问的气质。

**逐项查下来，没有一项「必须」问**——每个设置都有合理默认：存储初始化（静默）、傍晚阈值（默认 19:00，PUSH 已降级成一句提醒，误触发无害，v2 还会自动学）、自动提醒（默认开）、语言（§8：跟随数据/读 locale，不让选）、口吻（有默认）。**所以最好的 onboarding 是几乎没有 onboarding。**

**onboarding 真正要做的唯一一件事：一次性「轻框定」契约，当作第一次反思来交付。**

第一次特殊只因为：① 历史库空，第一题只能来自 seeds；② 用户不知道契约，尤其「答完不回应」不说明会显得冷。所以 onboarding = 一段 3~4 句温柔框定，说清四件事后**同一口气直接抛第一题**：

> 我会在你每天收工时，问你一个关于你自己的问题。
> 你答完，我安静收下——不评判、不建议、不啰嗦。一天就一次。
> 这些问题会顺着你的回答长，用得越久越懂你。
> 你的回答只存在你这台电脑上，永不上传。（想改提醒时间或口吻，随时说一声。）
>
> 那么，今天第一个问题——〈seed 第一题〉

要点：
- **配置靠默认，不靠提问**；自定义用「告知，不询问」一句带过（escape hatch = 自然语言或轻量 `/设置`，绝不做 wizard）。
- **第一题是 onboarding 的 UX**：别用「今天过得怎么样」泛题，要一上来就显出向内、苏格拉底质感。
- **不做历史预热问卷**（别一上来问「最近在想什么」凑 themes）——耐心等它自然累积，这份耐心本身就是产品。
- **第一答的收尾保持神圣**（「记住了，明天见」），**不在那一刻塞任何设置问题**（这是否决 School B 的理由：把神圣收尾让给沉默，比省一次「改时间」更重要）。

**触发面处理**：新装后首次**自动提醒**保持极轻（「Vesper 装好了，今晚想试试吗？`/收工` 开始」，不塞整段 intro）；真正的 intro + 第一题放在用户**主动 pull 第一次**时跑。检测：`~/.vesper/.onboarded` 标记不存在 → 走 intro → 写标记。

## 8. 语言策略：单一通用 skill，不做两套

**结论：做一套通用 skill，语言是运行时细节，不是两个版本。**

- **拆解后没有一样需要 fork 代码**：触发机制 / 本地存储 / 检索 / schema 全部语言无关；真正随语言变的（问题、「收下」语、镜子归纳、触发短语）要么是 LLM 运行时生成，要么是一个文案变量。做两套 = 维护翻倍、数据分裂、对外门面碎片化。
- **语言跟随用户数据自动切**：用户用什么语言答，历史回答库就是什么语言，RAG 喂回去模型自然用同语言问。语言「长在数据里」，与「用得越久越懂你」是同一机制，用户无需选择。
- **唯一分两份的是数据、不是代码**：中文 / 英文各一份**种子问题集**（仅冷启动 1~7 天用到），挂在同一套 skill 上。
- **分层**：skill 内核单一通用；种子集中英两份数据文件；对外门面（README / 示例）面向开发者受众，英文优先 / 双语。
- **冷启动语言判定**：读 agent locale，或首题用双语种子，之后跟随用户回答语言。
