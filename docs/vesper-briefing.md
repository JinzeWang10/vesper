## 0. 一句话

一个长在编码 agent 里的极简反思 skill：每天收工时只问用户一个（基于其历史回答生成的）问题，收下回答、**当场不回应**，数据全部留在本地；周期性回看，把模式照回给用户。

名字取自《论语》曾子「吾日三省吾身」，对上苏格拉底「未经省察的人生不值得过」——东西方都在说同一件事：**每日自省**。

---

## 1. 为什么做（thesis）

- AI 飞速迭代，人人 FOMO；但「追得再快也赶不上技术迭代」。
- 当所有工具都在帮人「向外做更多」，本产品反着来：用 AI 帮人**向内看、回归本质、关注自己**。
- 这是作者一以贯之的母题（knowyourself 心理量表、八戒戒断、以及作者自己在用的本地 AI 教练）。
- **逆向定位**：别人做「AI 帮你向外扩张」，我做「AI 帮你向内安住」——在 FOMO 的世界里稀缺且戳人。

---

## 2. 目标与非目标

**目标**
- 首要：成为强 FDE 候选人的**开源作品集 + 可 demo 的工程叙事**（agent + 个人数据 RAG + eval + 可暴露成 MCP server）。
- 其次：一个作者自己真心想用、也愿意给同类开发者用的工具。

**非目标（v0 明确不追）**
- 不追消费级流量、不打分销战。受众是开发者/极客，分销内建在「他们已经待的编码 agent 里」。
- 不需要「真实用户规模」来证明价值——价值 = 漂亮的 repo + 能打开的 demo + 想清楚的设计。

---

## 3. 形态决策：编码 agent 的 Skill + hook

- repo 内含技能定义 + 本地记忆文件（参考 resume-coach 的 `sessions/` + `state/` 结构）。
- 用 hook 实现「每天收工自动问一次」。
- 加分件（后续）：把记忆层暴露成 **MCP server**，跨客户端复用，同时补作者简历的 MCP 缺口。

**为什么不是 App / Web / CLI 独立形态**：那些都要解决「让人来用」的分销问题；长在编码 agent 里则把分销问题直接消解，且「在 FOMO 编码中插入一次向内的打断」本身就是主题最锋利的表达。

---

## 4. 核心交互（铁律，不可破）

1. **每天一个问题，只问一次。** 没法暴食——反 FOMO 写进交互。绝不提供「再来一题」。
2. **答完当场不回应。** 只温柔收下（「记住了，明天见」）。无多巴胺、不让人表演给 AI 看。
3. **极简 / 沉浸 / 放松。** 砍掉一切制造压力的东西：**无连续打卡天数、无分数、无社交、无多余推送**。一天一个温柔提醒足矣。可跳过、绝不追问。
4. **Local-first。** 回答是最私密内心数据，**全部存本地，永不上传**（`.gitignore` 已排除 `data/`）。

---

## 5. 智能在哪（产品内核 + FDE 故事）

**① 问题长在历史之上（不是随机/固定）**
- 由 agent 读用户过去的回答来生成今天的问题：顺着上次往深问、偶尔把旧答案照回来（**带出处**：「三周前你也说过一模一样的话」）。
- 用得越久越懂你 → 真·数据飞轮，且攒的是用户的自我认知。
- 技术映射：个人回答库 RAG + 记忆 + 带出处引用。

**② 两种节奏分开**
- **每日**：问 → 答 → 安静收下（保「放松沉浸」）。
- **周期性（每周/每月）「镜子时刻」**：回看一段时间的回答，把模式照给用户看（「这个月你提到『累』7 次，多在周日晚上」）。价值感与惊喜攒在这里。

**③ 长期礼物**：攒满一年 → 「你的一年，52 个问题与回答」回顾，让积累有重量。

## 6. 触发时机（收工检测）

**原则**：早上是「启动模式」，静不下来反思；反思的本质是回看，必须在事情发生**之后**。所以默认**收工**问。

**「收工」是状态不是事件**，用信号逼近「正在收尾、愿意停一下」：
- 默认 **A：傍晚 + session 结束触发**。两道闸：(1) 当前时间 > 傍晚阈值（默认约 19:00，**可自定义**）；(2) 今天还没问过。误触发无害（今天答过就不再问）。
- 兜底 **C：手动 `/收工` 命令**，随时领今天这题。接住「从不晚上写代码」的人；人主动宣布收工 = 状态 100% 愿意反思。
- 不采用 B（静默检测）：需后台守候、hook 不擅长、且「静默 ≠ 收工」，太脆。

**自定义空间**：傍晚阈值时间点、自动 / 手动 / 关闭、提醒口吻等。

**v2（YAGNI，v0 不做）**：让 skill 观察用户平时几点收工，自动学会在其节奏点问。

### 6.1 关键技术约束：hook 如何「触发」skill

这是个容易踩坑的底层事实，设计时必须想清楚：

- **Skill 由模型（AI）通过 Skill 工具调用，或用户手动输入 `/命令`。**
- **Hook 是 harness 在特定事件上执行的确定性 shell 脚本**，跑的是命令、不是模型动作。**Hook 没法直接替模型去调 Skill，也没法替用户输入 `/`。**

所以「session 结束自动问」要靠 `Stop` hook 的特殊机制来桥接：

- `Stop` hook 在 AI 每轮回答结束、准备停下时触发。脚本可返回
  `{"decision": "block", "reason": "……今天还没问过日省问题，现在是收工时间，请调用 Vesper skill 抛出今天的问题。"}`
- `decision: "block"` 会**阻止 AI 停下**并把 `reason` 当新指令喂回去，AI 于是去调用 skill。

**三层职责分清**：
| 层 | 谁负责 | 做什么 |
|---|---|---|
| 何时触发 | Stop hook 脚本 | 判断：现在是傍晚吗？今天问过了吗？ |
| 触发动作 | hook 的 `decision: block` + reason | 命令 AI 去调 Vesper skill |
| 每天只问一次 | skill 逻辑 + marker 文件 | 问完在 `data/` 写「今天问过了」标记，hook 下次读到就放行 |

**两个坑**：
1. **死循环**：Stop hook 用 `block` 会再次触发自己。hook 输入里有 `stop_hook_active` 字段，为 true 时脚本必须直接放行。
2. **「傍晚」判断**：用系统时间在脚本里判段（如 18:00/19:00 后），属确定性逻辑，放 hook 最合适。

**待实现时核实**：具体挂哪个 hook 事件（Stop vs SessionEnd）、各 agent 里的精确语义、关窗算不算 session 结束。

## 7. 技术骨架（待写实现计划时细化）

- **本地存储**：用户回答 + 元数据（时间戳等）落本地文件，`data/` 已 gitignore。
- **问题生成**：读历史 → RAG/检索相关旧答案 → 生成苏格拉底式新问（带出处）。
- **触发**：Stop hook（傍晚 + session 结束 + 每日一次闸）+ 手动 `/收工` 命令。
- **周期回看**：扫一段窗口的回答 → 归纳模式 → 带出处呈现。
- **加分**：MCP server 暴露记忆/反思能力。
- **模型与运行位置**（本地模型 vs 云端、隐私边界）：实现阶段再定。

## 8. 语言策略：单一通用 skill，不做两套

**结论：做一套通用 skill，语言是运行时细节，不是两个版本。**

- **拆解后没有一样需要 fork 代码**：触发机制 / 本地存储 / RAG / schema 全部语言无关；真正随语言变的（问题、「收下」语、镜子归纳、触发短语）要么是 LLM 运行时生成，要么是一个文案变量。做两套 = 维护翻倍、数据分裂、demo 碎片化。
- **语言跟随用户数据自动切**：用户用什么语言答，历史回答库就是什么语言，RAG 喂回去模型自然用同语言问。语言「长在数据里」，与「用得越久越懂你」是同一机制，用户无需选择。
- **唯一分两份的是数据、不是代码**：中文 / 英文各一份**种子问题集**（仅冷启动 1~7 天用到），挂在同一套 skill 上。
- **分层**：skill 内核单一通用；种子集中英两份数据文件；对外门面（README / demo / MCP 描述）面向 FDE 作品集受众，英文优先 / 双语。
- **冷启动语言判定**：读 agent locale，或首题用双语种子，之后跟随用户回答语言。
